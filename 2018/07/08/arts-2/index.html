Review

这周看了下有关字符集合和字符编码的文章。

The history of Character Encoding

关于字符编码，你所需要知道的

字符集就是计算机可正常显示的字符集合。从刚开始的 ASCII（26 个大小写英文字母和一些常见的符号）到 GBK（覆盖了所有简体字以及一部分其他字符）到现在的 Unicode（涵盖了目前人类使用的所有字符）。

在 Unicode 出现之前，所有的字符集都是和具体编码方案绑定在一起的，都是直接将字符和最终字节流绑定死了，例如 ASCII 编码系统规定使用 7 比特来编码 ASCII 字符集；GB2312 以及 GBK 字符集，限定了使用最多 2 个字节来编码所有字符，并且规定了字节序。这样的编码系统通常用简单的查表，也就是通过代码页就可以直接将字符映射为存储设备上的字节流了。

而 Unicode 将字符集和字符编码方案分开。具体的字符编码根据所采用的编码方案而定。也就是说，虽然每个字符在 Unicode 字符集中都能找到唯一确定的编号（字符码，又称 Unicode 码），但是决定最终字节流的却是具体的字符编码。例如同样是对 Unicode 字符 “A” 进行编码，UTF-8 字符编码得到的字节流是 0x41，而 UTF-16（大端模式）得到的是 0x00 0x41。

Tip

最近在实现一个需要后台给前端发送通知的功能，参考了http如何像tcp一样实时的收消息？的方案。

原理是当线程阻塞时不占用系统资源，这样通过把线程挂起，让 HTTP 请求超时，从而减轻 HTTP 轮询对服务器造成的压力。

这里打算采用阻塞队列的方式，选用了 Redis 中的 BRPOP/BLPOP 来实现。

当前端请求进来的时候，从 Redis 中通过 BRPOP/BLPOP 获取数据，如果没有数据则一直阻塞着直到 Redis 中有了数据或者超过等待时间。前端在接口返回数据或者请求超时后再次发起请求。这样就通过 HTTP 长轮询可以保证消息的实时性，并且这样的连接对服务的请求压力是超时时间/次。

Share

我觉得开发人员一定要多了解业务，不仅限于自家公司，也要多了解其他行业。了解业务对程序的设计和开发起到了很大的帮助。PRD 往往是很表面的东西，是抽象的业务加上各种限制条件后最终的呈现。所以开发人员能够从 PRD 中把这个抽象的业务理出来是很重要的，了解业务能帮助开发人员从 PRD 中把抽象业务逻辑提取出来。

为什么要找到抽象的业务？是为了底层逻辑的通用性，让系统有更好的灵活性和适用性。有了一个通用性的底层，每次业务逻辑的变更只是上层限制条件的变更。当然不可能无限大的通用，必须有一个边界限定。

一切脱离实际的设计都是扯淡。从业务出发，能知道系统设计重点和取舍。系统是为业务服务的，如果一个系统与业务不符，那就没有什么价值。
